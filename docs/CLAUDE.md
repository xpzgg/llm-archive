# docs/ — 下游排障文档

本目录存放面向下游同事的排障指南，目标是让他们**照着文档能自己定位问题，不需要 escalation**。

## 行文结构

所有文档统一按以下结构组织，不要跳过或调换顺序：

1. **一句话结论** — 先告诉读者结论是什么，不要铺垫
2. **背景** — 解释必要的机制和原理。读者不了解内核，所以要先让他们看懂"正常情况下是怎么回事"，再去理解"哪里出了问题"。不要堆术语，用通俗的语言把因果链讲清楚
3. **排查地图** — 一个总览流程图/决策树，按关键线索（如设备号、错误码）分支，标注每个分支对应哪个章节。读者看完地图就知道该往哪走
4. **问题分析和操作步骤** — 对应地图中每个节点展开，写清楚：报错长什么样、可能的原因、怎么确认、怎么修。用地图中的章节号/锚点关联起来

## 写作原则

- **面向不了解内核的读者**。不要假设他们知道什么是 module、什么是设备号，该解释就解释
- **可操作**。每个排查步骤都要给具体的命令，不要只说"检查一下 XXX"
- **先机制后问题**。让读者先理解正常流程，再讲哪里会断、为什么会断
- **用流程图/决策树导航**。读者应该能快速定位到自己的场景，不用通读全文
