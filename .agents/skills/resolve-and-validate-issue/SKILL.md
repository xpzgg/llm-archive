# Resolve and Validate Issue

围绕证据推进，目标是证明“问题为何发生、修复为何有效”，而不是完成一组固定动作。

## 持续迭代

把下面的阶段视为反馈闭环，而不是一次性流水线。任何不符合预期的输出都是新证据：找到它推翻的最早假设，更新复现条件、根因判断或修复方案，再重新执行后续验证。持续推进，直到问题被可靠解决并形成证据闭环；只有确实无法继续时，才带着明确的阻塞原因和已验证事实停下，不能把失败、未复现或不完整验证包装成完成。

## 建立基线

明确期望与实际行为，确认被测试的版本、环境和输入。先在未修改的基线上复现并保留可重复的证据；根据问题选择本机、容器、虚拟机或真实硬件等合适环境。若无法复现，先查清环境、版本、路径和触发条件的差异，不要据此宣称问题不存在或已经修复。

## 定位根因

从现象向前还原因果链，找到真正被破坏的不变量。按问题需要分析数据、状态、控制流、并发或生命周期，区分已证明事实、合理推断和未知条件。遇到证据与假设矛盾时，优先修正假设；不要把最终报错位置自动当成根因。

## 实施修复

修复根因对应的不变量，选择范围清晰且与现有设计一致的改动。评估相关版本、依赖和语义差异，避免只掩盖症状或引入更宽泛的行为变化。

## 验证结果

在尽量相同的条件下做基线与修复版本的 A/B 对照：基线表现出问题，修复版本不再出现。根据问题的确定性重复运行，并补充针对性测试和必要的回归检查。插桩、故障注入或简化模型可以辅助分析，但不能替代原始问题的真实复现。无法形成完整动态闭环时，明确说明验证边界。

## 退出标准

只有满足以下条件，才允许把任务表述为“完成”“已解决”或等价结论：

1. 已在未修复基线上复现问题，或已经穷尽可行复现路径并给出具体阻塞证据。
2. 已在修复版本上用尽量相同的环境、输入和触发路径做 A/B 对照。
3. 已证明修复命中了根因不变量，而不是只改变最终报错点。
4. 已记录关键证据：版本/commit、配置、命令、输入、日志或失败输出、修复前后结果。
5. 若动态验证未完成，最终交付必须明确标记为“补丁草案/静态验证完成，动态闭环未完成”，并说明还缺什么；不得包装成完成。

## Linux kernel crash 额外退出标准

处理 Linux kernel crash、KASAN/KCSAN/KMSAN、syzbot/syzkaller 报告、panic/oops 或运行期内核回归时，静态检查和对象编译不构成完整验证。除非用户明确只要求分析或补丁草案，否则必须执行以下动态验证：

1. 构建可启动的测试内核，启用问题所需的子系统、调试选项和复现选项。
2. 在真实硬件、QEMU、kvmtool 或其他 VM 中启动该内核；若没有现成环境，先检查并尝试准备 QEMU、kernel image、rootfs/initramfs，而不是直接以“当前容器不是目标内核”作为边界。x86-64 QEMU 环境可参考 syzkaller 的 Ubuntu host/QEMU VM/kernel setup 文档：https://github.com/google/syzkaller/blob/master/docs/linux/setup_ubuntu-host_qemu-vm_x86-64-kernel.md
3. 在 guest 内运行 PoC、selftest 或原始 reproducer，捕获串口日志、dmesg、返回码和必要的 sysfs/debugfs 状态。
4. 对故障注入类复现，证明注入确实命中目标路径，例如计数器归零、日志出现、返回值符合预期或目标错误路径被执行。
5. 在未修复基线和修复版本之间做 A/B：基线必须表现出原问题或同一根因导致的可观察失败；修复版本必须在同一触发条件下不再失败。
6. 如果 QEMU/硬件动态验证确实不可行，必须列出已经检查过的环境条件和精确阻塞点，并停止在“未完成动态验证”的状态；不得给出完成结论。

最终简洁交付：结论、复现方式、根因、修复理由、补丁前后证据，以及仍未确认的部分。
