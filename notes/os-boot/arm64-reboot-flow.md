# Linux arm64 reboot：旧系统如何安全地交出控制权

> 本文基于 Linux `v7.1-rc1`，但重点不是记忆当前版本的函数调用，
> 而是理解 reboot 背后的问题、解决思路和设计权衡。

## 1. 核心问题

Reboot 不是让 CPU 重新跳到内核入口那么简单。

运行中的系统里，很多参与者都能独立改变硬件或内存：

- 用户进程正在修改数据；
- 设备可能仍在执行 DMA；
- 中断和定时器仍会驱动内核代码；
- 其他 CPU 仍在运行旧内核；
- 平台固件还没有接管机器。

如果直接启动新系统，就可能出现“新旧两个世界同时操作同一台机器”：旧设备继续写入已被新内核复用的内存，旧 CPU 继续执行代码，文件系统也可能处于不一致状态。

因此，reboot 的本质是一次**系统控制权的有序移交**：

```text
旧系统停止产生新活动
  -> 等待或终止已有活动
  -> 逐层放弃对设备和 CPU 的控制
  -> 将最终控制权交给固件
  -> 固件复位整个平台
```

整个过程可以概括为两步：

```text
Quiesce：让旧系统安静下来
Reset：让平台重新进入确定的初始状态
```

Linux 主要负责 Quiesce，固件和硬件负责最终的 Reset。

## 2. 为什么必须分阶段关闭

Reboot 期间，内核的运行能力在不断收缩：

```text
完整运行环境
  -> 用户态退出
  -> 普通设备关闭
  -> 系统核心设施关闭
  -> 其他 CPU 停止
  -> 当前 CPU 只剩下触发 reset 的能力
```

越靠后，可用的服务越少，操作越不可逆。因此关闭顺序必须遵循一个原则：

> 先关闭依赖者，再关闭被依赖者；先做复杂清理，再执行底层复位。

例如，驱动需要中断、时钟和总线才能停止设备，所以必须先关闭设备，再关闭这些基础设施。若顺序相反，后续清理就会依赖已经不存在的运行环境。

Linux 将不同层次的工作分开，而不是让一个函数包办全部关闭：

| 层次 | 负责的问题 |
|---|---|
| 用户态 | 停止业务、同步数据、卸载文件系统 |
| 通用内核 | 协调子系统并关闭设备 |
| arm64 架构层 | 关闭中断并停止其他 CPU |
| 固件/硬件 | 复位所有 CPU 和 SoC 组件 |

这种分层让每一层只处理自己真正理解的资源，同时也带来约束：一旦进入下一阶段，就不能再依赖上一阶段已经关闭的能力。

## 3. 总体流程

正常 reboot 的主线如下：

```text
用户态准备关机
  -> kernel_restart()             组织内核关闭过程
     -> device_shutdown()         让普通设备停止活动
     -> syscore_shutdown()        关闭系统核心设施
     -> machine_restart()         进入 arm64 最终阶段
        -> smp_send_stop()        停止其他 CPU
        -> restart handler        选择平台复位方法
           -> PSCI                请求固件复位系统
```

函数名只是源码入口。真正需要记住的是状态变化：

```text
停止软件活动
  -> 停止设备活动
  -> 停止内核基础设施
  -> 停止并发执行
  -> 平台复位
```

## 4. 用户态与内核的分工

用户态知道哪些服务正在运行、哪些数据需要持久化；内核知道如何控制设备、CPU 和固件。因此正常 reboot 分为两个责任域：

```text
用户态：结束业务世界
内核态：结束硬件执行世界
```

`reboot` 系统调用本身不等于文件系统 `sync`。直接进入系统调用仍可以让机器复位，但无法替代用户态的服务退出和数据收尾。

所有正常重启请求最终汇入 `kernel_restart()`，由统一路径组织关闭顺序。这避免不同调用者各自实现一套不完整的 reboot 流程。

**So What：** Linux 能保证的是内核和硬件层面的有序关闭，而不是自动理解所有应用数据语义。系统可靠重启依赖用户态与内核态共同完成各自负责的部分。

## 5. `device_shutdown()`：为什么停 CPU 之前要先停设备

设备不是 CPU 的被动附属物。CPU 启动 DMA 后，即使驱动不再运行，设备仍可能继续读写内存：

```text
旧内核启动 DMA
  -> CPU 进入 reboot
  -> 新内核复用旧内存
  -> 旧设备继续写入
  -> 新系统被破坏
```

所以 `device_shutdown()` 的核心目标不是完整释放软件对象，而是让硬件进入 quiescent 状态，例如停止 I/O、DMA、中断和硬件队列。

这也是 `shutdown()` 与 `remove()` 的根本区别：

| 回调 | 目标 |
|---|---|
| `shutdown()` | 阻止硬件继续活动，为整机重启做准备 |
| `remove()` | 完整解除驱动绑定并回收资源 |

机器即将复位时，释放每一块内核内存没有太大意义；确保设备不再访问内存却非常重要。

### 设备依赖决定关闭顺序

设备之间通常存在依赖：

```text
网卡 -> PCIe 控制器 -> 时钟/电源域
```

网卡需要通过 PCIe 访问寄存器才能停止 DMA。如果先关闭 PCIe 控制器，网卡就失去了完成 shutdown 的条件。

因此启动和关闭方向相反：

```text
启动：supplier/parent -> consumer/child
关闭：consumer/child -> supplier/parent
```

driver core 先冻结新的设备 probe，再依据父子关系、注册顺序和 device link 反向遍历设备。全局顺序由 driver core 管理，总线共同规则由 bus 层管理，具体硬件操作由 driver 完成：

```text
driver core：决定谁先关闭
bus：处理一类设备的共同要求
driver：停止具体硬件
```

**So What：** 这种设计把硬件差异留给驱动，同时让内核统一管理依赖顺序。但它只能处理内核已知的依赖；驱动若没有正确描述 supplier/consumer 关系，关闭顺序就可能出错。

对于普通 reboot，最终硬件 reset 可能停止所有设备；对于 kexec，新内核不经过完整硬件复位就直接接管机器，因此正确停止 DMA 尤其关键。

## 6. 为什么还需要 `syscore_shutdown()`

并非所有组件都适合当作普通设备关闭。中断控制器、timer、clocksource 等设施本身支撑着内核运行，过早关闭会让普通设备失去完成 shutdown 所需的环境。

因此 Linux 把它们放在普通设备之后处理：

```text
先关闭使用系统服务的设备
再关闭提供系统服务的核心组件
```

**So What：** 普通 device model 无法自然表达所有底层系统资源，`syscore` 提供了一个最后阶段的补充机制。代价是进入该阶段后，内核能够依赖的基础服务进一步减少，后续代码必须非常克制。

## 7. arm64：把多 CPU 系统收缩为最后一个执行者

普通设备和核心设施关闭后，arm64 的 `machine_restart()` 会关闭当前 CPU 的本地中断，并通过 `smp_send_stop()` 请求其他 CPU 停止。

为什么需要先停其他 CPU？因为只复位当前 CPU 不代表旧系统已经消失。其他 CPU 仍可能运行旧内核并访问 RAM，从而与启动中的新系统冲突。

软件停核通常通过 IPI 完成，但它只能是 best effort：某个 CPU 可能卡在无法响应 IPI 的状态。因此二者职责不同：

```text
软件停核：尽量在 reset 前终止旧内核并发
平台 reset：最终保证所有 CPU 和必要组件一起复位
```

**So What：** 软件机制改善关闭过程的可控性，却不能取代系统级硬件复位。arm64 平台必须提供能够覆盖整机的 reset，否则 Linux 无法仅靠软件证明旧系统已经完全停止。

## 8. PSCI：为什么由固件完成最终复位

完整复位可能涉及 CPU cluster、电源域、内存控制器和 SoC 外设。Linux 未必拥有操作这些资源的权限，也不应该包含每一种 SoC 的私有复位知识。

PSCI 为 arm64 提供统一的固件接口：

```text
Linux EL1
  -> PSCI SYSTEM_RESET
  -> SMC
  -> EL3 firmware
  -> SoC reset controller
```

虚拟机中则可能通过 HVC 把请求交给 EL2 hypervisor。

通用内核使用 restart handler 抽象 PSCI、watchdog、PMIC、syscon 等不同复位后端。Linux 负责表达“我要系统复位”，具体如何操作硬件由平台实现决定。

**So What：** 这种分工降低了 Linux 对 SoC 的耦合，并把高权限的平台操作留给固件；代价是 reboot 的可靠性依赖固件质量。若 PSCI 或 EL3 实现有问题，内核通常无法绕过固件自行完成等价的整机复位。

成功的 reset 调用不会返回。如果所有复位方法都失败，arm64 只能停在原地，因为此时系统已经被部分关闭，继续运行比停止更危险。

## 9. 正常重启与紧急重启的权衡

正常路径追求有序关闭：

```text
通知子系统
  -> 停止设备
  -> 关闭核心设施
  -> 停止其他 CPU
  -> 平台复位
```

但 panic 时内核状态可能已经损坏，锁、调度或设备回调都不再可信。继续追求完整清理，可能让机器永久卡死。

因此 `emergency_restart()` 会跳过大部分清理，尽快请求平台复位：

```text
正常重启：优先保证一致性
紧急重启：优先保证可恢复性
```

**So What：** Linux 没有一条在所有故障状态下都最优的 reboot 路径。清理越完整，对当前内核可正常运行的依赖越强；路径越短，越可能成功复位，但数据损坏和设备未停止的风险也越高。

## 10. 用系统模型定位 reboot 卡死

排查时先判断控制权移交停在哪一层，而不是立刻钻进具体函数。

### 卡在某个设备 shutdown

系统仍处于 Quiesce 阶段。重点检查：设备是否还有未结束的 I/O、shutdown 是否等待不会再发生的事件，以及 consumer/supplier 的关闭顺序是否正确。

### 已打印 `Restarting system`，但机器没有复位

普通设备和大部分核心组件通常已经关闭。问题更可能位于其他 CPU 停止、EFI/restart handler、PSCI、EL3 固件或 SoC reset controller。

### 提示 secondary CPU 无法停止

说明旧内核的并发执行没有完全结束。需要沿 stop IPI、GIC、CPU 异常状态和固件协作方向排查；但最终仍应确认平台 reset 是否能够强制复位所有 CPU。

## 11. 源码阅读入口

按控制权逐层收缩的顺序阅读：

```text
kernel/reboot.c
  kernel_restart()

drivers/base/core.c
  device_shutdown()

arch/arm64/kernel/process.c
  machine_restart()

arch/arm64/kernel/smp.c
  smp_send_stop()

drivers/firmware/psci/psci.c
  psci_sys_reset()
```

阅读代码时始终围绕四个问题：

- 这一层负责什么？
- 它要消除哪个仍在活动的参与者？
- 它采用什么思路解决问题？
- 这种设计获得了什么，又付出了什么代价？

最终需要建立的认知不是一条函数调用链，而是一个稳定的设计模型：

> Reboot 是旧系统逐层放弃能力和控制权，最终由固件把整个平台带回确定状态的过程。
