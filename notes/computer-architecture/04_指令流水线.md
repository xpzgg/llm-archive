# 指令流水线

## 基本概念

### 五个流水线阶段

```
取指 → 译码 → 执行 → 访存 → 写回
```

每条指令必须经历这五个阶段才算完成。

### 单周期处理器

每条指令在一个时钟周期内走完全部五个阶段。

**时钟周期 = 所有阶段耗时之和**（比如 9ns）

**存在意义**：设计最简单，没有中间状态——中间产生的临时信号只是导线上的电压（组合逻辑），从不被触发器锁存，时钟边沿后自然消失。一个周期结束只保留 PC 和寄存器堆。

**问题**：时钟周期由最慢阶段决定，其他阶段空转浪费。

### 流水线处理器

在每两个阶段之间插入**流水线寄存器**，不同指令同时处于不同阶段：

```
周期：   1    2    3    4    5    6
指令A:  取指  译码  执行  访存  写回
指令B:        取指  译码  执行  访存  写回
指令C:              取指  译码  执行  访存  写回
```

**时钟周期 = 最慢单个阶段耗时**（比如 3ns），理想 CPI = 1。

**关键**：流水线寄存器不只存数据，还存控制信号，随指令逐级传递。

**本质**：把串行变并行，用空间（寄存器）换时间（吞吐量）。和土木工程的流水施工/流水节拍是同一个道理。

注意：流水线不减少单条指令的延迟，只提高吞吐量。

**容易混淆：单周期和流水线的 CPI 都是1，但含义不同**
```
单周期：1条指令 × 9ns/周期 = 9ns
流水线：1条指令 × 3ns/周期 = 3ns
```
CPI 只是指令数与周期数的比值，不反映周期长度。真正衡量性能要看三者乘积：指令数 × CPI × 时钟周期长度。

**"中间状态消失"的物理原因**：不是主动清除，而是根本没存过。取指、译码、执行等阶段都是组合逻辑，输入变输出立刻跟着变，没有记忆。只有被触发器锁存的信号才有状态，单周期处理器的中间信号从未被锁存，时钟边沿后随新输入自然消失。

## 性能指标

```
CPI（Cycles Per Instruction）= 每条指令平均周期数
IPC = 1/CPI（现代处理器常用）

执行时间 = 指令数 × CPI × 时钟周期长度

流水线CPI = 理想CPI + 数据冲突气泡 + 控制冲突气泡 + 结构冲突气泡
```

## 三类流水线冲突

### 数据相关冲突

**三种类型**：
- RAW（写后读）：真依赖，后面指令用前面指令还没写完的结果 → 会冲突
- WAW（写后写）：假依赖，简单流水线不冲突，乱序执行时需处理
- WAR（读后写）：假依赖，简单流水线不冲突，乱序执行时需处理

**解决方案**：
1. **阻塞**：插入气泡等待，简单但低效
2. **前递**：执行结果通过额外线路直接传给下一条指令，不等写回

**Load-use hazard**：load 指令数据在访存阶段才有，而下一条指令的执行阶段同时到达，前递来不及，必须强制等一个周期插气泡。编译器可在 load 后插入不相关指令填充气泡。

### 控制相关冲突

跳转目标要到执行阶段才知道，但流水线已经取了后面的指令进来，取错了要作废。

**解决方案**：
1. **阻塞**：检测到跳转指令后插气泡
2. **提前计算**：把跳转判断提前到译码阶段，只等1个周期
3. **延迟槽**：ISA 规定跳转后一条指令无论如何都执行，编译器填有用指令，硬件0开销
4. **分支预测**：提前猜方向，预测错误再冲刷

### 结构相关冲突

多条指令竞争同一硬件资源。解决：阻塞，或增加硬件资源。

## 提高流水线效率的技术

### 多发射

每个周期同时发射多条指令，流水线"变宽"，理想 CPI 降到 0.5（双发射）甚至更低。

代价：硬件翻倍，同周期指令间的依赖检测更复杂，存在"陪同阻塞"（两条指令必须同时推进）。

**与普通流水线的区别**：
- 流水线：不同指令同时处于不同阶段（纵向并行）
- 多发射：多条指令同时处于同一阶段（横向并行）

### 动态调度（乱序执行）

**核心思想**：前面指令在等数据时，后面没有依赖的指令先执行，挖掘指令级并行。

**两个关键结构**：
- **保留站**：指令进来先入队等待，源操作数全部就绪后立刻乱序发射执行
- **重排序缓冲（ROB）**：指令乱序执行，但必须按原始顺序提交结果（保证精确异常和程序语义）

```
取指 → 译码 → 进保留站等待 → 乱序执行 → 结果进ROB → 顺序提交
              ↑ 乱序开始                              ↑ 有序结束
```

**寄存器重命名**：把逻辑寄存器映射到不同物理寄存器，消除 WAW/WAR 假依赖，让乱序执行更自由。

```
逻辑寄存器（ISA定义，程序员可见，数量少）
     ↓ 重命名表（动态映射）
物理寄存器（硬件实体，数量多，如192个）
```

Tomasulo 算法（1966，IBM 360/91）是硬件寄存器重命名的鼻祖。

### 转移预测

转移指令平均每 5-10 条出现一次，多发射处理器中预测错误代价极大（冲刷十几条指令）。

**BTB（分支目标缓冲）**：记录跳转目标地址，取指阶段就能预测跳去哪里。

**PHT（模式历史表）**：预测跳还是不跳。
- 1位 PHT：记录上次结果，单层循环准确率90%，两层嵌套循环降至80%
- 2位 PHT（饱和计数器）：连续两次预测错才改变方向，两层循环提升至88%

```
计数器：  0    1  |  2    3
预测：   不跳  不跳 | 跳   跳
```

### Cache（高速缓存）

**Why**：CPU 周期 < 1ns，内存访问 ~50ns，差距50倍以上。利用局部性原理在 CPU 旁放小而快的存储。

**存储层次**：
```
寄存器 < 1ns → L1 Cache ~1ns → L2 Cache ~5ns → L3 Cache ~20ns → 内存 ~50ns → 磁盘
```

**三种组织方式**：
- 直接相联：每个内存块只能放一个位置，查找快，容易冲突
- 全相联：可放任意位置，冲突少，硬件复杂
- 组相联：折中，N路，每个块有N个候选位置

**地址划分**：`高位Tag | 中间Index | 低位Offset`

**写策略**：
- 写穿透：每次写立刻同步到下层，简单但慢
- 写回：先写Cache，替换时才写回**下一层存储**（Cache写回内存，L1写回L2），快但复杂

**注意**："写回"有两个不同含义：
1. 流水线第五阶段"写回" = 把执行结果写回寄存器堆
2. Cache"写回策略" = 把Cache块写回下一层存储
两者是完全独立的概念。

## 精确异常

**挑战**：流水线里多条指令同时在飞，异常发生时需要保证"异常前指令全部完成，异常指令及之后未执行"。

**实现**：异常标记随指令流水传递到写回阶段才统一处理。写回阶段是指令真正修改机器状态的地方，在此之前都是"临时"状态。

## 触发器、锁存器、寄存器

- **锁存器**：电平触发，使能为1时透明，为0时锁存。时序难控制。
- **触发器**：两个锁存器串联，只在时钟边沿采样，边沿触发天然抵抗噪声。
- **寄存器**：N个触发器并联，共享时钟，存N位数据。

```
锁存器（电平触发，1位）→ 触发器（边沿触发，1位）→ 寄存器（边沿触发，N位）
```

各种"寄存器"名字（PC、流水线寄存器、控制寄存器等）本质都一样，区别只在用途和访问方式。

## 时钟频率与流水线深度的权衡

时钟周期下限由**关键路径**（最长组合逻辑延迟 + 触发器建立时间 + 时钟偏斜）决定。

流水线平衡目标：各阶段耗时均等，时钟周期压到最低。慢阶段可继续拆分（流水线加深）。

代价：流水线越深，分支预测失败惩罚越大。奔腾4（31级流水线）是反面教材。
