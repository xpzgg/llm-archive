# 并行编程基础

## 10.1 三种并行性

### 指令级并行（ILP）
单个程序内，无依赖的指令同时执行。由硬件自动挖掘（流水线、乱序执行），程序员感知不到。天花板受限于数据依赖链。

### 数据级并行（DLP）
对数组每个元素执行相同操作，元素间无依赖，天然可并行。需要 SIMD 硬件支持——一条指令同时处理多个数据。

### 任务级并行（TLP）
不同任务分配到不同处理单元，程序员显式管理。任务间需要通信和同步。

### 三者与 GPU 的关系
- SIMD：GPU 基础执行模式，一个 warp 所有线程执行相同指令
- SIMT：SIMD 的扩展，每个线程有自己的寄存器和 PC，允许分支，但同一 warp 内分支导致串行化（warp divergence）
- TLP：大量 warp 并发，隐藏访存延迟

## 10.2 并行编程模型

### 核心区别只有一条：地址空间是否共享

地址空间共享 → 通信隐式（直接读写变量）、同步必须显式（加锁）
地址空间独立 → 通信显式（send/receive）、同步隐含在通信中、可伸缩性更好

其余所有差异（bug形态、编程复杂度等）都是这一条的推论。

### 数据并行模型的五个特性

1. **单线程视角**：程序员看到的是单一控制流
2. **同构并行**：所有处理单元执行相同操作，只是数据不同（对比TLP可以做不同的事）
3. **全局命名空间**：所有变量在单一地址空间，所有语句可访问任何变量
4. **隐式相互作用**：每条语句结束有隐含 barrier，通信通过变量赋值完成，程序员不需要写 sync
5. **隐式数据分配**：程序员只描述做什么，不需要指定数据分给谁处理

**Q：隐式同步的本质是什么？**

数据并行要求锁步执行（Lockstep）——所有处理单元像队伍齐步走，一条语句全部完成才进入下一条。这是"隐式 barrier"的物理实现。

**Q：锁步执行的代价是什么？**

遇到分支时，不同元素走不同路径，但锁步要求同步。解决方法是两个分支串行执行，走另一路的处理单元空转等待——并行度减半。这就是 GPU 的 warp divergence，是性能大敌。

### 归约问题（重要）

```c
a[i] = b[i] + c[i];   // 安全：每个处理单元写自己的 a[i]
result += d[i];         // 危险：N个处理单元同时写同一个 result
```

多个处理单元同时读-改-写同一变量，产生竞争条件，结果不确定。

**正确做法：归约树**

分层合并，N个元素只需 log₂N 步，每步内部并行：
```
d[0] d[1] d[2] d[3]
  └─+─┘     └─+─┘
    s01        s23
      └───+───┘
         result
```

数据并行模型的"隐式相互作用"只对 map 类操作（每人写自己的输出）成立，归约是边界情况，必须显式处理。

对应到 CUDA：`atomicAdd` 是最简单的解法但串行化所有写操作，真正高效的做法是归约树 + shared memory。

### SMP vs NUMA

两者都是共享存储的硬件实现：

**SMP（对称多处理）**：所有处理器访问任意内存延迟相同。核数少时可实现，核数多时内存总线成瓶颈。

**NUMA（非均匀内存访问）**：每个 CPU 有本地内存，访问本地快（~10ns），跨节点慢（~40ns）。核数多时的现实选择，现代多路服务器几乎都是 NUMA。

**对编程的影响**：代码层面看起来一样（都是共享地址空间），但 NUMA 下不感知拓扑的代码性能会莫名变差。感知 NUMA 的优化：把数据分配在使用它的 CPU 的本地内存，线程绑定到固定 CPU（CPU affinity）。

## 10.3 典型并行编程环境

### Pthreads

共享存储模型最底层的 C 接口，所有并行逻辑全靠手写。

核心同步原语：

**pthread_mutex**：保护临界区，防止多个线程同时访问共享数据。语义："我在用，你别进来。"

**pthread_cond_wait**：等待某个条件成立，条件不满足时挂起线程。语义："等某件事发生，先睡觉，有消息叫我。"

**Q：mutex 保护的是 cond 变量本身吗？**

不是。mutex 保护的是**条件判断所依赖的共享状态**（比如 queue 这个数据结构）。cond 只是通知机制，完全不知道条件是什么，只提供"给我一个睡觉的地方"和"叫醒一个在这里睡觉的人"两个原语。

条件（queue_empty）和条件变量（cond）是完全分离的，cond 只是一个等待队列。

**Q：为什么 cond_wait 需要配合 mutex，等待期间 mutex 是否释放？**

等待期间 mutex 释放，醒来后自动重新持有。cond_wait 原子地执行：
1. 释放 mutex（让生产者能进来修改状态）
2. 挂起线程
3. 被唤醒后重新获取 mutex

步骤1和2必须原子，否则会错过 signal：
```
消费者：检查队列为空
                      ← 生产者插入数据，发出通知（没人收到）
消费者：挂起等待      ← 永远等不到了
```

**Q：被唤醒后为什么要用 while 而不是 if 重新检查条件？**

两个原因：
1. 虚假唤醒（Spurious Wakeup）：POSIX 允许 cond_wait 在没有 signal 时自己醒来
2. 多消费者竞争：多人同时被唤醒，只有一个能拿到数据，其他人条件又不满足了

**Q：醒来后 mutex 是自动拿到的还是要竞争？**

是去竞争，不是自动拿到。cond_wait 内部第三步"重新获取 mutex"会阻塞直到拿到锁。多个线程被 broadcast 唤醒时，所有人都去竞争同一把锁，只有一个拿到，其余继续阻塞等 mutex。

这也是为什么必须用 while——B 等到 mutex 后，A 可能已经把数据消费掉了，B 必须重新检查条件。

**Q：重新获取 mutex 的代码写在哪里？**

藏在 `pthread_cond_wait` 内部，不需要手动写。从调用方视角：
```c
pthread_mutex_lock(&mutex);       // 手动拿锁
while (queue_empty()) {
    pthread_cond_wait(&cond, &mutex);
    // 进去时：持有 mutex，内部释放
    // 出来时：内部重新拿回，继续持有
}
queue_pop();
pthread_mutex_unlock(&mutex);     // 手动释放锁
```
锁的释放和重新获取全部封装在 cond_wait 里，对外像一个原子操作。

**Q：futex requeue 优化的原理**

broadcast 的问题（惊群）：
```
唤醒所有线程 → 所有人竞争 mutex → 只有1个拿到 → 其余进内核态睡眠
→ 大量无意义的用户态/内核态切换
```

futex requeue 优化：
```
broadcast 时只唤醒1个线程
其余线程直接从 cond 的 futex 等待队列"搬移"到 mutex 的 futex 等待队列
这些线程没有被唤醒过，一直在睡
等 mutex 释放后，由 mutex 的逻辑依次唤醒
```

这个优化能做的前提是：重新获取 mutex 的逻辑封装在 cond_wait 内部，glibc 知道"醒来后下一步一定是抢 mutex"，所以可以在内核里直接做搬移，省掉一次"唤醒→拿不到锁→再睡"的往返。

**Q：为什么 cond 可以 broadcast 唤醒所有人，而 semaphore 只唤醒一个？**

本质区别在于语义：

- semaphore 是**资源**：sem_post 释放一个资源，只有一个人能拿到，唤醒多个没意义
- cond 是**条件通知**：broadcast = "状态变了，你们都来检查一下"，条件满不满足每个人自己判断（while 重新检查），唤醒所有人是合理的语义

requeue 优化的精妙：**保留了 broadcast 的语义**（所有人最终都会被唤醒检查），同时**避免了惊群的开销**（不是同时唤醒，而是排队挂在 mutex 上，一个处理完再唤醒下一个）。

**cond 的底层实现**：基于 futex，内部维护序列号。cond_wait 记录当前序列号，释放 mutex，调用 futex_wait 等序列号变化。futex_wait 的语义是"当前值不等于期望值时立刻返回"，消灭了 unlock 和 wait 之间的竞态窗口。

**信号量 vs 条件变量**

信号量内置计数器，有"记忆"——没人等时 sem_post，计数器+1，下次 sem_wait 直接通过。条件变量无记忆，没人等时 signal 直接丢失。

```
资源计数语义（连接池、停车位）→  semaphore
复杂条件等待（队列非空、状态机）→  cond_wait + mutex
```

### OpenMP

共享存储模型的高层封装，通过编译器指令（pragma）自动生成多线程代码：

```c
#pragma omp parallel for reduction(+:sum)
for (i = 0; i < N; i++) {
    sum += f(i);
}
```

**本质**：把手写 Pthreads 的样板代码（线程创建、任务分配、barrier、reduction 的 mutex 保护）用一行 pragma 代替。没有提供任何 Pthreads 做不到的能力，解决的是工程效率问题。

去掉 pragma 代码仍然是合法串行程序（向后兼容）。执行模型是 fork-join：主线程遇到并行区域 fork 出线程组，结束时 join 回来。

适合给已有串行代码快速加速；需要精细控制时还是要回到 Pthreads。

### MPI

消息传递模型的工业标准。MPI 本身是规范，有多种实现（OpenMPI、MPICH、IntelMPI）。

**Q：MPI 是接口协议还是可用框架？**

两者都是：MPI 是规范（定义函数签名和语义），具体实现有多个。类似 HTTP 标准和 Apache/nginx 的关系。程序写一份 MPI 代码，换不同实现只需重新链接库。

**Q：为什么 MPI 没有被取代？**

硬件厂商深度绑定（InfiniBand 针对 MPI 优化）、巨大的存量代码、性能接近硬件极限、可移植性极好。AI/ML 领域已被 NCCL+PyTorch Distributed 取代，传统 HPC 仍是 MPI 统治。

#### MPI 四种发送模式

核心问题：**发送方什么时候可以安全地修改发送缓冲区？**

两种根本解法：
- 等对方收到再返回（等待）
- 把数据拷一份立刻返回（拷贝）

四种模式的本质：
```
缓冲 Send（Bsend） →  强制走拷贝路线，程序员提供缓冲区，立刻返回
同步 Send（Ssend） →  强制走等待路线，等接收方开始调用 MPI_Recv 后才返回
标准 Send（Send）  →  MPI 自己决定用哪种（小消息拷贝，大消息等待）
就绪 Send（Rsend） →  等待的变体，跳过握手，假设接收方已经在等了
```

**Q："发完了"的定义各不相同**

- 缓冲 Send：数据拷进缓冲区了（最弱）
- 标准 Send：数据离开我的缓冲区了（可以覆盖了）
- 同步 Send：握手完成，传输可以开始了（不是数据到达）
- 就绪 Send：跳过握手，直接开始传

MPI 没有任何一种 Send 会等到数据完全到达才返回。

**Q：同步 Send 的"等接收方"是等数据到达吗？**

不是。是等双方完成握手协商，确认接收方准备好了，传输可以开始。数据在哪不确定，不一定完全到达。

#### Eager/Rendezvous 协议

标准 Send 内部的实现策略：

**小消息（Eager）**：直接拷进 MPI 内部缓冲发出，接收方到了放缓冲等取。
- 拷贝开销低，内存压力小，延迟低，不需要握手

**大消息（Rendezvous）**：先握手再传：
1. 发送方发一个小通知"我要发大数据"
2. 接收方调用 MPI_Recv，回复"我准备好了"
3. 发送方直接把数据传过去（零拷贝，直接内存到内存）

InfiniBand 上用 RDMA：网卡直接读本地内存写入接收方内存，全程不经过 CPU。

Trade-off：
```
小消息 Eager：   牺牲内存（多一次拷贝）  换取低延迟（不握手）
大消息 Rendezvous：牺牲延迟（握手往返）  换取零拷贝和内存安全
```

切换点（通常几十KB）是在延迟和内存开销之间找平衡。

#### MPI 可靠性

**Q：丢包、数据错误由 MPI 处理还是 TCP？**

MPI 标准只承诺"Send 和 Recv 都成功返回则数据正确完整"，不规定怎么做到。完全委托给底层传输层：
- 跑在 TCP 上：TCP 的校验和、重传、有序到达
- 跑在 InfiniBand 上：硬件级可靠性（RC Queue Pair），比 TCP 延迟更低

MPI 不处理节点挂掉的情况，默认行为是整个作业 abort。ULFM 扩展尝试解决这个问题，但还没进主标准。

#### 缓冲发送后谁来发

**Q：数据拷进缓冲区后，是谁负责实际发送？**

MPI 内部有**进度引擎（Progress Engine）**负责推进所有未完成的通信。触发时机：
- 后台通信线程持续驱动（常见实现）
- 调用任何 MPI 函数时顺带推进

**陷阱**：如果 MPI 实现没有后台线程，长时间做纯计算不调用任何 MPI 函数，缓冲区数据可能一直没发出去，缓冲区满了下次 Bsend 直接报错（不是阻塞等待）。

## OpenMP + MPI + CUDA 的关系

三者解决三个不同维度的并行：
```
OpenMP  →  单机多核 CPU（共享内存，任务级）
MPI     →  多机多节点（分布式，消息传递）
CUDA    →  单机 GPU（大量简单核心，数据级）
```

大型 HPC/AI 训练程序经常三者叠加：节点间 MPI，节点内 CPU 用 OpenMP，GPU 用 CUDA。
