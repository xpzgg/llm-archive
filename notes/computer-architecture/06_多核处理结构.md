# 多核处理结构

> 本章是概览，细节配合《What Every Programmer Should Know About Memory》（Ulrich Drepper）一起读。

## 本章三条主线

本章围绕三个递进的问题展开：

```
片上 Cache 如何组织
    ↓ 组织好了，多核共享同一内存，出现新问题
Cache 数据一致性如何维护
    ↓ 协议解决了硬件层面的一致性，但程序员视角还有问题
存储一致性模型是什么
```

- **Cache 组织**：物理结构问题——Cache 怎么摆，私有还是共享，几级，UCA 还是 NUCA
- **Cache 一致性**：由第一个引出——多核各有私有 Cache，同一地址有多份副本，一个核修改了其他核怎么知道？MESI、侦听/目录协议回答这个问题
- **存储一致性模型**：比第二个更抽象一层——即使硬件保证了 Cache 一致性，乱序执行、写缓冲、编译器重排仍然影响可见性，程序员视角"我写了一个值另一个核什么时候能看到"需要明确约定

三者关系：
```
Cache 组织    →  决定硬件结构
Cache 一致性  →  解决硬件层面副本同步
存储一致性模型 →  解决软件层面可见性约定
```

Drepper 那篇文章主要在第三个问题上深挖，教你怎么写出符合存储一致性模型的正确高效代码。

## 11.1 为什么出现多核

**Q：晶体管数量增加是同一功能用更多晶体管，还是实现更多功能？**

不是同一个功能用更多晶体管，而是多出来的晶体管可以用来实现原来没有的功能模块：

```
1980s：简单单发射流水线
1990s：多出来的晶体管 → 乱序执行、分支预测、多发射（挖掘 ILP）
2000s：多出来的晶体管 → 更大的 Cache、更深的流水线
2005s：多出来的晶体管 → 多个完整的核（ILP 收益递减，转向 TLP）
```

晶体管是"预算"，增加了就有钱实现更复杂的机制。早期用来买 ILP，后来 ILP 收益递减（功耗墙），就改买多核。之前学的所有流水线优化技术，都是在花这笔预算。

**功耗墙**：单核提频，功耗超线性增长（频率翻倍，功耗×8）。多核设计让"功耗随性能线性增长"，2005年后成为主流。

摩尔定律还在，晶体管还在增加，但不再用来提单核频率，而是堆核数。

## 11.2 访存结构

### 片上 Cache 组织

主流方案：L1/L2 私有（每个核独享），L3 共享（所有核共用）。

**UCA vs NUCA**：
- UCA（统一 Cache 访问）：集中式，所有核访问 L3 延迟相同，核数多了总线成瓶颈
- NUCA（非统一 Cache 访问）：分布式，L3 切成多块分布在芯片各处，靠近的核访问快，可扩展性好

### 存储一致性模型

多核共享内存时，"我写了一个值，另一个核能看到吗？什么时候看到？"这个问题的答案就是一致性模型。

| 模型 | 约束强度 | 性能 | 说明 |
|---|---|---|---|
| 顺序一致性（SC） | 最严格 | 最低 | 所有核看到的操作顺序完全一致 |
| 处理器一致性（PC） | 较宽松 | 中 | 允许 load 超越 store 执行 |
| 弱一致性（WC） | 宽松 | 高 | 区分同步操作和普通操作 |
| 释放一致性（RC） | 最宽松 | 最高 | 进一步细分 acquire 和 release |

越宽松性能越好，但程序员越难推理正确性。Java/C++ 的 memory order 就是在这些模型上的编程抽象。

### Cache 一致性协议

多个核各有私有 Cache，同一个内存地址可能被多个核缓存，写操作如何保持一致？

**两种策略**：
- 写无效（Write Invalidate）：我要写时，先把其他核的副本置为无效，我独占再写
- 写更新（Write Update）：我写时，把新值广播给所有持有副本的核

写无效更常用，因为写更新产生大量广播流量。

**两种协议实现**：
- 侦听协议（Snooping）：所有核监听共享总线上的消息，适合核数少的情况
- 目录协议（Directory）：用目录记录每个 Cache 行在哪些核有副本，只通知相关核，适合核数多（总线不够用）

**MESI 四个状态**：

```
M（Modified）：只有我有，且被修改过，内存是旧的
E（Exclusive）：只有我有，未修改，和内存一致
S（Shared）：多个核都有副本，未修改
I（Invalid）：无效，不能用
```

状态机驱动一致性：读/写操作触发状态转换，发消息给其他核。

## 11.3 互连结构

多个核之间、核和 Cache 之间怎么连：

| 方案 | 特点 | 适用 |
|---|---|---|
| 片上总线 | 简单，但带宽共享，核多了成瓶颈 | 4核以下 |
| 交叉开关 | 高带宽，任意核对互通，但面积随核数平方增长 | 中等规模 |
| 片上网络（NoC） | 可扩展，核数多时仍然高效，但延迟有路由开销 | 众核 |

现代高核数处理器（如 Intel 多核、龙芯）基本都用环形总线或 Mesh 网络。

## 11.4 同步机制

### 原子操作

多核下普通的读-改-写不是原子的，需要硬件支持：

**RMW 指令**（Test-and-Set、Compare-and-Swap、Fetch-and-Add）：一条指令完成读-改-写，硬件保证原子性。

**LL/SC 指令对**（Load-Linked / Store-Conditional）：
- LL：读值，同时标记"我在监视这个地址"
- SC：写值，如果这个地址没被别人动过才成功，否则失败重试
- RISC 架构（ARM、MIPS、RISC-V）常用这个，比 CAS 更灵活

### 自旋锁

最简单的锁：拿不到就一直循环等（忙等）：

```c
while (test_and_set(&lock));  // 拿到了才退出
// 临界区
lock = 0;
```

问题：多个核同时自旋，持续产生 Cache 一致性流量，总线压力大。改进方案：排队锁（每个核在自己的变量上自旋，减少总线广播）。

### 栅障（Barrier）

等所有核都到达某个点，才继续执行。集中式实现简单但可扩展性差，树形栅障更好。

### 事务内存（TM）

新型同步机制，像数据库事务一样：把一段代码标记为事务，硬件保证原子性和隔离性，冲突时自动回滚重试。代表：Intel TSX。

**Q：事务内存和锁的本质区别是什么？**

锁解决的不只是粒度问题，真正的问题是**多把锁组合使用时极难写对**：

```c
lock(&bucket[3]);
lock(&bucket[7]);    // 另一个线程反过来先锁7再锁3 → 死锁
move(3, 7);
unlock(&bucket[7]);
unlock(&bucket[3]);
```

多把锁的加锁顺序、异常处理、嵌套逻辑，复杂度随锁数量指数级上升。

事务内存让程序员只声明意图，不用思考锁的组合问题：

```c
__transaction_atomic {
    move(bucket[3], bucket[7]);  // 硬件保证原子，死锁从根本上不可能
}
```

**本质区别**：
```
锁：  悲观，有人在跑就等，程序员负责正确性
事务：乐观，大家都跑，提交时验证，硬件负责正确性
```

**回退的触发条件**：不是"发现有别人在跑"，而是"发现有别人访问了同一块数据"：

```
A 操作 bucket[3]，B 操作 bucket[7] → 无数据重叠 → 都提交成功
A 操作 bucket[3]，B 也操作 bucket[3] → 数据重叠 → 一个回退重试
```

两者从结果上等价（都保证原子性和隔离性），但实现路径不同：
- 冲突少：事务赢（大家都能跑完，不用排队）
- 冲突多：锁赢（事务反复回退浪费）

## 11.5 几个典型处理器

| 处理器 | 特点 |
|---|---|
| 龙芯 3A5000 | 4核，分布式共享 L3，目录协议 |
| Intel SandyBridge | 环形互连，侦听协议，集成 GPU |
| IBM Cell | 异构：1个主核 + 8个 SPE，用局部存储代替 Cache，程序员手动管理数据搬移 |
| NVIDIA Fermi | 众核 GPU，512个核，Warp 调度 |

IBM Cell 是个极端例子——完全不用 Cache 一致性协议，直接让程序员用 DMA 搬数据，性能极高但编程极难，最终没有普及。
