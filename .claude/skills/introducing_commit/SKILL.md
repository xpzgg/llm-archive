# 引入问题补丁分析

在本地内核源码中追溯漏洞代码的引入 commit，给出扎实的论证。
**所有回复使用中文，技术术语保留英文原文。**

---

## 前置条件

需要以下信息之一：
- 当前会话中已完成 CVE 分析（有根因分析和修复逻辑分析的结果）
- 用户直接提供了 patch diff + 漏洞特征描述

若信息不足，明确告知用户缺少哪些信息。

---

## 分析原则

### 起点：从前文分析中提取漏洞特征

不要盲目搜索。先从 CVE 分析上下文（commit message、patch diff、根因分析、修复逻辑）中提取漏洞的核心特征，作为追溯的起点：
- 关键变量名（如 `num_buffers`、`watch_id`）
- 缺失的检查逻辑（如无边界校验、无类型检查）
- 错误的比较基准（如 `> end` 而非 `> end - addr`）
- 设计缺陷的本质（如 signed/unsigned 不匹配、锁协议遗漏）

这些特征决定了用哪些关键词在 git 历史中搜索。

### 目标

找到**真正引入漏洞逻辑**的 commit，而非仅引入了函数框架或无关改动的 commit。

### 启发式策略

根据漏洞类型和可用信息灵活组合，不强固定序：

- **利用 `Fixes:` 标签**：如果 patch 本身标注了 `Fixes:`，这就是引入 commit，只需验证
- **从漏洞特征出发追溯**：基于提取的漏洞特征，在 git 历史中搜索引入该特征（关键变量、错误逻辑、缺失检查）的 commit
- **区分框架 vs 漏洞**：引入 commit 应包含漏洞代码本身。如果 commit 只是加了个空函数或 stub，后来另一个 commit 才填入了有问题的逻辑，后者才是引入 commit
- **多候选时缩小范围**：如果追溯出多个可能的引入 commit，用更精确的漏洞特征排除不相关的

### 验证

最终结论必须经过验证：
- 引入 commit 之前，该漏洞不可触发（没有触发路径、有其他保护机制、或相关代码根本不存在）
- 论证基于 git 的实际输出，不是猜测

---

## 输出格式

```
引入 Commit：<hash> ("<title>")
论证：<简述论证过程，说明为什么是这个 commit 而不是其他 commit>
验证：<确认引入前缺陷不可触发，以及具体依据>
```

---

## 约束

- 结论基于 git 的实际输出，不臆测
- 无法确定唯一引入 commit 时，列出候选并说明各自的依据和疑点
- 漏洞从内核初始版本就存在时，如实说明
